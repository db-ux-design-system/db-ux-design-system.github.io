###### Use the right control

Use Input for free text entry. If users must choose from predefined options, use a selection component instead (e.g., Select, Autocomplete, Radio/Checkbox depending on the case).

###### Always provide a visible label

Every Input must have a clear, descriptive label that explains what to enter. Keep labels short and specific; avoid vague labels. Do not replace the label with a placeholder.

###### Use placeholder text only as an example

Placeholder text is optional and should provide an example format/value, not instructions or required context. Placeholders must not contain critical information because they disappear when users type.

Use helper text to prevent errors (format, constraints, expectations). Use validation messages to explain what is wrong and how to fix it. Keep messages concise and actionable.

Ensure every Input has an accessible name that matches the visible label. If a label is visually hidden, provide an equivalent screenreader label. Required/optional indicators must be announced meaningfully.

## FAQ

<DBAccordion>
	<DBAccordionItem headlinePlain="When should I use Input vs Textarea?">
		Use Input for single-line, short values. Use Textarea for multi-line, longer content (comments,
		descriptions, messages).
	</DBAccordionItem>
	<DBAccordionItem headlinePlain="Can I omit the label if the UI is compact?">
		No. Keep a visible label whenever possible. If space is extremely constrained, use a visually
		hidden label but keep the accessible name intact.
	</DBAccordionItem>
	<DBAccordionItem headlinePlain="Should validation happen while typing or on submit?">
		Prefer validating on blur or on submit for most fields to reduce distraction. Use inline,
		real-time validation only when it clearly prevents errors (e.g., format constraints) and
		messages remain calm and actionable.
	</DBAccordionItem>
	<DBAccordionItem headlinePlain="What should I show for required vs optional fields?">
		Follow the system rule consistently: either mark required fields (common) or mark optional
		fields (common when most are required). In all cases, ensure the requirement is conveyed to
		screenreaders and reflected in validation messaging.
	</DBAccordionItem>
</DBAccordion>
