## Guidelines

### Use Checkbox for multi-select, not single-choice

Use checkboxes when users can select any number of options (including none). If the user must pick exactly one option, use Radio instead.

### Write short, positive labels

Labels must be clear and concise. Prefer positive phrasing to reduce cognitive load and avoid double negatives.

### Make groups easy to scan

In checkbox groups, keep options parallel in wording and length. If a group needs context, add a group label or short description above it.

## FAQ

<DBAccordion>
	<DBAccordionItem headlinePlain="When should I use Checkbox vs Switch?">
		Use Checkbox for selections in forms, lists, and filters (often multiple at once). Use Switch
		for a single on/off setting that takes effect immediately.
	</DBAccordionItem>
	<DBAccordionItem headlinePlain="When should I use Checkbox vs Radio?">
		Use Checkbox when multiple options can be selected. Use Radio when exactly one option must be
		chosen within a set.
	</DBAccordionItem>
	<DBAccordionItem headlinePlain="Can I hide the visible label?">
		Only if you provide equivalent screenreader text and the meaning is still clear in context.
		Prefer visible labels whenever possible.
	</DBAccordionItem>
	<DBAccordionItem headlinePlain="How should I validate checkbox groups?">
		If at least one option is required, validate the group (not each item) and show a clear error
		message that explains what is missing.
	</DBAccordionItem>
	<DBAccordionItem headlinePlain='Do I need a "Select all" option?'>
		Only when it saves effort for large groups and users commonly select most options. Keep the
		state accurate and support indeterminate for partial selection.
	</DBAccordionItem>
</DBAccordion>
