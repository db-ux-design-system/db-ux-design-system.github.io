<DBSection spacing="small" width="auto">
The DB UX Design System follows the principles of **Semantic Versioning** (SemVer): version numbers consist of three segments — `MAJOR.MINOR.PATCH` — and clearly communicate the nature and scope of a change.

</DBSection>

<DBNotification headline="Versioning across Design and Development" variant='standalone' semantic='informational'>

Major and Minor version numbers are kept in sync across Design and Development. Patches are tracked separately. If only one side (Design or Dev) introduces a change, the shared version number is still incremented for both to maintain consistency.

</DBNotification>

<DBSection spacing="small" width="auto">

## Major

A Major release introduces **breaking changes** — changes to the API or Design Library that are not backward-compatible. Consumers should expect to adapt their implementations to the new API.

<DBNotification headline="Cadence" variant='standalone' showIcon={false}>

Major releases follow a planned schedule with at least 2–3 months between releases. The planned scope is announced in advance in the \*\*DB UX Design System Teams channel.

</DBNotification>

**Examples:**

- Overhaul of the color structure or visual identity
- Renaming of component settings
- New components that alter existing behavior

## Minor

A Minor release introduces **new features and improvements** without changing the API or existing functionality — fully backward-compatible.

<DBNotification headline="Cadence" variant='standalone' showIcon={false}>

Minor releases align with the team's two-week sprint cycle but may be skipped if no relevant changes are ready. Once a month, a summary post is published in the Teams channel, covering all releases from the preceding weeks.

</DBNotification>

**Examples:**

- Enhancements to existing components
- New components without breaking changes

## Patch

A Patch release contains **bug fixes or very small improvements** with no impact on the API or functionality. Patches are published outside the regular cadence as needed.

**Examples:**

- A bug fix for a component that has no impact on other features

</DBSection>
