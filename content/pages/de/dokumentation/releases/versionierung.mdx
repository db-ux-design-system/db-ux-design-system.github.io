<DBSection spacing="small" width="auto">
Das DB UX Design System folgt den Prinzipien des **Semantic Versioning** (SemVer): Versionsnummern bestehen aus drei Segmenten – `MAJOR.MINOR.PATCH` – und kommunizieren klar Art und Umfang einer Änderung.

</DBSection>

<DBNotification headline="Versionierung über Design und Entwicklung hinweg" variant='standalone' semantic='informational'>

Major- und Minor-Versionsnummern werden zwischen Design und Entwicklung synchron gehalten. Patches werden separat geführt. Wenn nur eine Seite (Design oder Entwicklung) eine Änderung einführt, wird die gemeinsame Versionsnummer trotzdem für beide erhöht, um Konsistenz zu gewährleisten.

</DBNotification>

<DBSection spacing="small" width="auto">

## Major

Ein Major Release führt **Breaking Changes** ein – Änderungen an der API oder Design-Library, die nicht abwärtskompatibel sind. Nutzer:innen sollten damit rechnen, ihre Implementierungen an die neue API anzupassen.

<DBNotification headline="Kadenz" variant='standalone' showIcon={false}>

Major Releases folgen einem geplanten Zeitplan mit mindestens 2–3 Monaten zwischen den Releases. Der geplante Umfang wird vorab im DB UX Design System Teams-Kanal angekündigt.

</DBNotification>

**Beispiele:**

- Überarbeitung der Farbstruktur oder visuellen Identität
- Umbenennung von Komponenten-Einstellungen
- Neue Komponenten, die bestehendes Verhalten verändern

## Minor

Ein Minor Release führt **neue Features und Verbesserungen** ein, ohne die API oder bestehende Funktionalität zu ändern – vollständig abwärtskompatibel.

<DBNotification headline="Kadenz" variant='standalone' showIcon={false}>

Minor Releases orientieren sich am zweiwöchigen Sprint-Zyklus des Teams, können aber übersprungen werden, wenn keine relevanten Änderungen bereitstehen. Einmal im Monat wird ein Zusammenfassungspost im Teams-Kanal veröffentlicht, der alle Releases der vorangegangenen Wochen abdeckt.

</DBNotification>

**Beispiele:**

- Erweiterungen bestehender Komponenten
- Neue Komponenten ohne Breaking Changes

## Patch

Ein Patch Release enthält **Bugfixes oder sehr kleine Verbesserungen** ohne Auswirkungen auf API oder Funktionalität. Patches werden bei Bedarf außerhalb der regulären Kadenz veröffentlicht.

**Beispiele:**

- Ein Bugfix für eine Komponente ohne Auswirkungen auf andere Features

</DBSection>
